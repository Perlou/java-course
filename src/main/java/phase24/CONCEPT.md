## ⭐ 维度建模

### 星型模型

```
               ┌─────────────┐
               │  时间维度    │
               │  dim_date   │
               └──────┬──────┘
                      │
┌─────────────┐       │       ┌─────────────┐
│  商品维度    │       │       │  用户维度    │
│ dim_product │───────┼───────│  dim_user   │
└─────────────┘       │       └─────────────┘
                      │
               ┌──────┴──────┐
               │   事实表     │
               │ fact_order  │
               │ (订单事实)   │
               └──────┬──────┘
                      │
               ┌──────┴──────┐
               │  地区维度    │
               │ dim_region  │
               └─────────────┘

特点：一个事实表关联多个维度表
优点：查询简单，性能好
```

### 雪花模型

```
┌───────────┐    ┌───────────┐
│  品类     │───→│  商品     │
└───────────┘    └─────┬─────┘
                       │
                       ▼
                ┌──────────────┐
                │    事实表     │
                └──────────────┘

特点：维度表进一步规范化
优点：减少冗余
缺点：查询复杂，需要更多 Join
```

## 🔄 缓慢变化维度 (SCD)

| 类型     | 策略     | 说明                 |
| -------- | -------- | -------------------- |
| **SCD1** | 覆盖更新 | 直接覆盖，不保留历史 |
| **SCD2** | 新增行   | 保留所有历史版本     |
| **SCD3** | 新增列   | 保留有限历史         |

```
SCD2 示例（用户地址变更）
═════════════════════════════════════════════════════════
user_id | city   | start_date | end_date   | is_current
--------|--------|------------|------------|------------
1001    | 北京   | 2020-01-01 | 2023-06-30 | 0
1001    | 上海   | 2023-07-01 | 9999-12-31 | 1
```

| 技术               | 特点                       | 适用场景   |

### Lakehouse 架构

```
┌─────────────────────────────────────────────────────────┐
│                   Lakehouse 架构                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│   传统架构                     Lakehouse 架构            │
│                                                         │
│   ┌──────┐    ┌──────┐       ┌──────────────────────┐  │
│   │数据湖 │ + │数据仓 │  →    │      Lakehouse       │  │
│   │(存储) │   │(分析) │       │  统一存储 + 分析      │  │
│   └──────┘    └──────┘       └──────────────────────┘  │
│                                                         │
│   优势：                                                │
│   • 统一存储，减少数据冗余                              │
│   • 支持 ACID 事务                                      │
│   • 支持流批一体                                        │
│   • 开放格式，多引擎访问                                │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## 📊 数据治理

### 核心领域

| 领域           | 内容                   | 工具               |
| -------------- | ---------------------- | ------------------ |
| **数据质量**   | 完整性、准确性、一致性 | Great Expectations |
| **元数据管理** | 技术/业务元数据        | Apache Atlas       |
| **数据血缘**   | 数据来源与流向         | Datahub            |
| **数据安全**   | 访问控制、脱敏         | Apache Ranger      |

### 数据质量规则

```
数据质量检查维度
═════════════════════════════════════════════════════════

完整性  → 非空检查、必填字段
准确性  → 格式校验、范围检查
一致性  → 跨表一致、业务规则
及时性  → 数据新鲜度、更新频率
唯一性  → 主键唯一、去重检查
```

## 📝 Hive 核心

### 存储格式对比

| 格式     | 类型 | 压缩率 | 查询性能 |
| -------- | ---- | ------ | -------- |
| TextFile | 行式 | 低     | 低       |
| ORC      | 列式 | 高     | 高       |
| Parquet  | 列式 | 高     | 高       |

### 分区与分桶

```sql
-- 分区表（按日期分区）
CREATE TABLE orders (
    order_id BIGINT,
    amount DECIMAL
) PARTITIONED BY (dt STRING);

-- 分桶表（按用户 ID 分桶）
CREATE TABLE users (
    user_id BIGINT,
    name STRING
) CLUSTERED BY (user_id) INTO 32 BUCKETS;
```
