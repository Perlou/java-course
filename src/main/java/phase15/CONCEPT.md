## 📊 CAP 定理

### 什么是 CAP？

```
┌─────────────────────────────────────────────────────────────────────┐
│                          CAP 定理                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│        分布式系统最多只能同时满足以下三项中的两项：                  │
│                                                                     │
│                         Consistency                                 │
│                         (一致性)                                    │
│                            ▲                                        │
│                           / \                                       │
│                          /   \                                      │
│                         /     \                                     │
│                        /   C   \                                    │
│                       /_________\                                   │
│                      /           \                                  │
│                     / CA      CP  \                                 │
│                    /               \                                │
│    Availability ◀─────── P ─────────▶ Partition                     │
│     (可用性)                            Tolerance                   │
│                                        (分区容错)                    │
│                                                                     │
│  ⚠️ 在分布式系统中，P 是必须的，所以实际选择是 CP 或 AP             │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### CAP 选择

| 选择   | 描述              | 典型系统                     | 适用场景           |
| ------ | ----------------- | ---------------------------- | ------------------ |
| **CP** | 一致性 + 分区容错 | ZooKeeper, etcd, HBase       | 金融交易、库存扣减 |
| **AP** | 可用性 + 分区容错 | Cassandra, DynamoDB, CouchDB | 社交动态、评论系统 |
| CA     | 一致性 + 可用性   | 传统单机数据库               | 不适用分布式       |

## 📊 BASE 理论

### CAP 的补充

```
BASE 是对 CAP 中 AP 的延伸，追求最终一致性

┌─────────────────────────────────────────────────────────────────┐
│                         BASE 理论                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  BA - Basically Available (基本可用)                            │
│  ────────────────────────────────────────────────────────────── │
│  • 允许响应时间略有增加                                          │
│  • 允许功能降级（如：高峰期引导页）                               │
│                                                                 │
│  S - Soft State (软状态)                                        │
│  ────────────────────────────────────────────────────────────── │
│  • 允许数据存在中间状态                                          │
│  • 不同节点的数据副本不一致是允许的                              │
│                                                                 │
│  E - Eventually Consistent (最终一致性)                         │
│  ────────────────────────────────────────────────────────────── │
│  • 经过一段时间后，所有数据最终达成一致                          │
│  • 不保证实时一致，但保证最终一致                                │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### ACID vs BASE

| ACID                 | BASE                            |
| -------------------- | ------------------------------- |
| 原子性 (Atomicity)   | 基本可用 (Basically Available)  |
| 一致性 (Consistency) | 软状态 (Soft State)             |
| 隔离性 (Isolation)   | 最终一致 (Eventual Consistency) |
| 持久性 (Durability)  |                                 |
| **强一致性**         | **弱一致性**                    |
| **悲观策略**         | **乐观策略**                    |

## 🏗️ 架构模式

### 分层架构

```
┌─────────────────────────────────────────────────────────────────┐
│                         分层架构                                  │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  表现层 (Presentation)                   │   │
│  │               Controller, View, DTO                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  业务层 (Business/Service)               │   │
│  │                Service, Domain Logic                     │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  持久层 (Persistence/DAO)                │   │
│  │                  Repository, Mapper                      │   │
│  └─────────────────────────────────────────────────────────┘   │
│                              ↓                                  │
│  ┌─────────────────────────────────────────────────────────┐   │
│  │                  数据层 (Database)                       │   │
│  └─────────────────────────────────────────────────────────┘   │
│                                                                 │
│  ✅ 优点：简单直观、职责分明                                     │
│  ❌ 缺点：业务逻辑容易泄露、难以测试                              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 六边形架构 (端口与适配器)

```
┌─────────────────────────────────────────────────────────────────┐
│                         六边形架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│         适配器                                    适配器         │
│    ┌───────────┐                              ┌───────────┐    │
│    │   REST    │                              │   MySQL   │    │
│    │   API     │                              │  Adapter  │    │
│    └─────┬─────┘                              └─────┬─────┘    │
│          │                                          │          │
│          ↓                                          ↓          │
│    ┌───────────┐     ┌─────────────────┐     ┌───────────┐    │
│    │  入站端口  │ ──→ │                 │ ──→ │  出站端口  │    │
│    │  (接口)   │     │  领域核心        │     │   (接口)   │    │
│    └───────────┘     │  Domain Core    │     └───────────┘    │
│          ↑           │                 │           ↑          │
│          │           └─────────────────┘           │          │
│    ┌─────┴─────┐                              ┌─────┴─────┐    │
│    │   gRPC    │                              │   Redis   │    │
│    │  Adapter  │                              │  Adapter  │    │
│    └───────────┘                              └───────────┘    │
│                                                                 │
│  ✅ 优点：                                                      │
│  • 领域核心与外部隔离                                           │
│  • 技术细节可替换（换数据库只需换适配器）                        │
│  • 易于测试（Mock 端口即可）                                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 事件驱动架构

```
┌─────────────────────────────────────────────────────────────────┐
│                       事件驱动架构                                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│    生产者                 事件总线                  消费者       │
│                                                                 │
│  ┌─────────┐          ┌───────────────┐         ┌─────────┐   │
│  │ 订单服务 │ ──────→  │               │ ──────→ │ 库存服务 │   │
│  └─────────┘          │   Event Bus   │         └─────────┘   │
│                       │  (Kafka/RMQ)  │                       │
│  ┌─────────┐          │               │         ┌─────────┐   │
│  │ 支付服务 │ ──────→  │               │ ──────→ │ 通知服务 │   │
│  └─────────┘          └───────────────┘         └─────────┘   │
│                                                                 │
│  事件类型：                                                     │
│  • 领域事件：OrderCreated, PaymentCompleted                     │
│  • 集成事件：跨服务通信                                         │
│                                                                 │
│  ✅ 优点：松耦合、可扩展、异步处理                               │
│  ❌ 缺点：最终一致性、调试复杂、事件顺序                         │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 📐 DDD 领域驱动设计

### 战略设计

```
┌─────────────────────────────────────────────────────────────────┐
│                      限界上下文 (Bounded Context)                │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│   电商系统领域划分                                               │
│                                                                 │
│   ┌─────────────┐   ┌─────────────┐   ┌─────────────┐          │
│   │   订单域     │   │   商品域     │   │   用户域     │          │
│   │             │   │             │   │             │          │
│   │  Order      │   │  Product    │   │  User       │          │
│   │  OrderItem  │   │  SKU        │   │  Address    │          │
│   │  Payment    │   │  Category   │   │  Account    │          │
│   └─────────────┘   └─────────────┘   └─────────────┘          │
│         │                 │                 │                  │
│         └────────────────┼─────────────────┘                  │
│                          ↓                                     │
│             ┌─────────────────────────────┐                    │
│             │   上下文映射 Context Map     │                    │
│             │   • 共享内核                 │                    │
│             │   • 防腐层                   │                    │
│             │   • 开放主机服务             │                    │
│             └─────────────────────────────┘                    │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 战术设计

| 概念                          | 描述               | 示例               |
| ----------------------------- | ------------------ | ------------------ |
| **实体 (Entity)**             | 有唯一标识，可变   | Order, User        |
| **值对象 (Value Object)**     | 无标识，不可变     | Money, Address     |
| **聚合 (Aggregate)**          | 一组相关对象的集合 | Order + OrderItems |
| **聚合根 (Aggregate Root)**   | 聚合的入口         | Order              |
| **领域服务 (Domain Service)** | 无状态的业务逻辑   | TransferService    |
| **领域事件 (Domain Event)**   | 领域中发生的事件   | OrderCreated       |
| **仓储 (Repository)**         | 聚合的持久化       | OrderRepository    |

## 🔄 CQRS 与事件溯源

### CQRS 模式

```
┌─────────────────────────────────────────────────────────────────┐
│              CQRS (Command Query Responsibility Segregation)     │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│              ┌─────────────────────────────────┐                │
│              │           客户端                 │                │
│              └───────────┬─────────────────────┘                │
│                          │                                      │
│            ┌─────────────┴─────────────┐                       │
│            ↓                           ↓                       │
│    ┌───────────────┐           ┌───────────────┐               │
│    │   Command     │           │     Query     │               │
│    │   命令处理     │           │    查询处理    │               │
│    └───────┬───────┘           └───────┬───────┘               │
│            ↓                           ↓                       │
│    ┌───────────────┐           ┌───────────────┐               │
│    │  Write Model  │ ────────→ │  Read Model   │               │
│    │   写模型       │   同步    │   读模型       │               │
│    │  (PostgreSQL) │           │   (Redis)     │               │
│    └───────────────┘           └───────────────┘               │
│                                                                 │
│  ✅ 优点：读写分离优化、独立扩展                                 │
│  ❌ 缺点：复杂度增加、数据同步                                   │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 事件溯源

```
┌─────────────────────────────────────────────────────────────────┐
│                      Event Sourcing                              │
├─────────────────────────────────────────────────────────────────┤
│                                                                 │
│  传统方式：存储当前状态                                          │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │ Order: { id: 1, status: "PAID", total: 100 }              │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
│  事件溯源：存储事件序列                                          │
│  ┌───────────────────────────────────────────────────────────┐ │
│  │ Event 1: OrderCreated { orderId: 1, items: [...] }        │ │
│  │ Event 2: ItemAdded { orderId: 1, productId: 100 }         │ │
│  │ Event 3: PaymentReceived { orderId: 1, amount: 100 }      │ │
│  └───────────────────────────────────────────────────────────┘ │
│                                                                 │
│  状态重建: 回放所有事件 → 得到当前状态                           │
│                                                                 │
│  ✅ 优点：完整历史记录、可审计、时间旅行                         │
│  ❌ 缺点：复杂度高、存储量大、查询困难                           │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```
