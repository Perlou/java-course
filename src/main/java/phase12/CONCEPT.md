## 🎯 Redis 数据结构

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Redis 五种数据类型                              │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  String (字符串)                                                    │
│  ├── SET key value / GET key                                       │
│  ├── INCR/DECR (原子计数器)                                        │
│  └── 适用: 缓存、计数器、分布式锁                                  │
│                                                                     │
│  Hash (哈希)                                                        │

│                                                                     │
│  List (列表)                                                        │

│  └── 适用: 消息队列、最新列表                                      │
│                                                                     │
│  Set (集合)                                                         │
│  ├── SADD / SMEMBERS / SINTER (交集)                               │
│  └── 适用: 标签、共同好友、去重                                    │
│                                                                     │
│  ZSet (有序集合)                                                    │
│  ├── ZADD key score member / ZRANGE                                │
│  └── 适用: 排行榜、延迟队列                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 💥 缓存问题与解决

```
┌─────────────────────────────────────────────────────────────────────┐
│                      缓存三大问题                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  缓存穿透 (查询不存在的数据)                                        │
│  ──────────────────────────────────────────────                     │
│  问题: 查询 DB 中不存在的数据，缓存无法命中                         │
│  解决:                                                              │
│    • 缓存空值 (TTL 设短)                                            │
│    • 布隆过滤器 (Bloom Filter)                                      │
│                                                                     │
│  缓存击穿 (热点 key 失效)                                           │
│  ──────────────────────────────────────────────                     │
│  问题: 热点 key 过期，大量请求打到 DB                               │
│  解决:                                                              │
│    • 分布式锁 (只一个请求查 DB)                                     │
│    • 逻辑过期 (不设 TTL，后台异步更新)                              │
│                                                                     │
│  缓存雪崩 (大量 key 同时失效)                                       │
│  ──────────────────────────────────────────────                     │
│  问题: 大量缓存同时过期，DB 压力剧增                                │
│  解决:                                                              │
│    • 随机过期时间 (TTL + random)                                    │
│    • 多级缓存 (本地 + Redis)                                        │
│    • 降级限流                                                       │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

### 缓存一致性

```
┌─────────────────────────────────────────────────────────────────────┐
│                      缓存更新策略                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Cache-Aside (旁路缓存) - 最常用                                    │
│  ──────────────────────────────────────────────                     │
│  读: 先读缓存 → 未命中 → 读 DB → 写缓存                             │
│  写: 先更新 DB → 再删除缓存                                         │
│                                                                     │
│  ⚠️ 为什么是删除而不是更新缓存?                                     │
│  • 避免并发写导致脏数据                                             │
│  • 延迟计算，按需加载                                               │
│                                                                     │
│  延迟双删 (应对并发问题)                                            │
│  ──────────────────────────────────────────────                     │
│  1. 删除缓存                                                        │
│  2. 更新 DB                                                         │
│  3. 延迟 N 毫秒后再删除缓存                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 🔐 分布式锁

```
┌─────────────────────────────────────────────────────────────────────┐
│                      分布式锁实现                                    │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  Redis 实现:                                                        │
│  ──────────────────────────────────────────────                     │
│  // 加锁 (SET NX EX)                                                │
│  SET lock:order:123 "uuid" NX EX 30                                │
│                                                                     │
│  // 释放锁 (Lua 保证原子性)                                         │
│  if redis.get("lock:order:123") == "uuid" then                     │
│      redis.del("lock:order:123")                                   │
│  end                                                                │
│                                                                     │
│  Redisson 实现 (推荐):                                              │
│  ──────────────────────────────────────────────                     │
│  RLock lock = redisson.getLock("lock:order");                      │
│  try {                                                              │
│      lock.lock();  // 自动续期                                      │
│      // 业务逻辑                                                    │
│  } finally {                                                        │
│      lock.unlock();                                                 │
│  }                                                                  │
│                                                                     │
│  特性:                                                              │
│  • 可重入锁                                                         │
│  • 看门狗自动续期 (30秒)                                            │
│  • 红锁 (RedLock) 多节点容错                                        │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 🔄 分布式事务

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Seata 事务模式                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  AT 模式 (自动补偿) - 默认推荐                                       │
│  ──────────────────────────────────────────────                     │
│  1. 一阶段: 执行 SQL，记录 undo log                                 │
│  2. 二阶段提交: 删除 undo log                                       │
│  2. 二阶段回滚: 根据 undo log 反向补偿                              │
│                                                                     │
│  TCC 模式 (显式控制)                                                │
│  ──────────────────────────────────────────────                     │
│  Try:     预留资源 (冻结库存)                                       │
│  Confirm: 确认提交 (扣减库存)                                       │
│  Cancel:  取消回滚 (释放库存)                                       │
│                                                                     │
│  Saga 模式 (长事务)                                                 │
│  ──────────────────────────────────────────────                     │
│  正向: T1 → T2 → T3                                                 │
│  补偿: C3 → C2 → C1                                                 │
│                                                                     │
│  选型建议:                                                          │
│  • AT: 大多数场景，无侵入                                           │
│  • TCC: 高一致性要求，需要编码                                      │
│  • Saga: 长流程，最终一致性                                         │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘
```

## 🆔 分布式 ID

```
┌─────────────────────────────────────────────────────────────────────┐
│                      分布式 ID 方案                                  │
├─────────────────────────────────────────────────────────────────────┤
│                                                                     │
│  UUID                                                               │
│  ├── 优点: 本地生成，无依赖                                         │
│  └── 缺点: 无序，索引效率低，不利于排序                             │
│                                                                     │
│  雪花算法 (Snowflake) - 推荐                                        │
│  ├── 64 位 = 符号位(1) + 时间戳(41) + 机器ID(10) + 序列号(12)       │
│  ├── 优点: 有序，高性能，包含时间信息                               │
│  └── 缺点: 依赖机器时钟，时钟回拨问题                               │
│                                                                     │
│  号段模式 (Leaf, 美团)                                              │
│  ├── 从 DB 批量获取号段 (如 1-1000)                                 │
│  ├── 本地消耗完再申请下一段                                         │
│  └── 优点: 高可用，批量减少 DB 压力                                 │
│                                                                     │
│  Redis INCR                                                         │
│  ├── 原子自增，简单高效                                             │
│  └── 缺点: 依赖 Redis，需要持久化                                   │
│                                                                     │
└─────────────────────────────────────────────────────────────────────┘

雪花算法结构:
┌───┬───────────────────────────────┬──────────┬────────────┐
│ 0 │        41 位时间戳            │ 10位机器 │ 12位序列号 │
└───┴───────────────────────────────┴──────────┴────────────┘
     约 69 年                         1024节点   4096/ms
```

## 📝 核心要点

1. **Redis**: 五种数据类型各有适用场景，ZSet 做排行榜
2. **缓存设计**: 穿透用布隆过滤器，击穿用分布式锁，雪崩用随机 TTL
3. **分布式锁**: 生产用 Redisson，自带续期和可重入
4. **消息队列**: RocketMQ 业务消息，Kafka 日志/大数据
5. **消息可靠**: 生产重试 + 消费幂等 + 死信兜底
6. **分布式事务**: Seata AT 无侵入，TCC 强一致
7. **分布式 ID**: 雪花算法是主流选择
