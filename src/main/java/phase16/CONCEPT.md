## 🎯 高并发的核心挑战

### 1. 什么是高并发？

高并发指系统在短时间内接收并处理大量请求的能力。衡量指标：

| 指标   | 英文                    | 描述         | 典型值          |
| ------ | ----------------------- | ------------ | --------------- |
| QPS    | Queries Per Second      | 每秒查询数   | 电商首页 10 万+ |
| TPS    | Transactions Per Second | 每秒事务数   | 支付系统 1 万+  |
| RT     | Response Time           | 响应时间     | < 200ms         |
| 并发数 | Concurrent Users        | 同时在线用户 | 百万级          |

### 2. 高并发三大问题

```
┌─────────────────────────────────────────────────────────────┐
│                    高并发三大问题                              │
├─────────────────┬─────────────────┬─────────────────────────┤
│   流量过载       │   资源争抢       │    故障雪崩              │
│  Overload       │  Contention     │   Avalanche            │
├─────────────────┼─────────────────┼─────────────────────────┤
│ 瞬间流量 > 容量  │ 多线程竞争资源    │ 一个服务故障引发连锁      │
│ 系统响应变慢     │ 数据库连接耗尽    │ 整个系统不可用           │
│ 甚至系统崩溃     │ 线程池打满       │ 级联失败                 │
├─────────────────┼─────────────────┼─────────────────────────┤
│ 解决: 限流       │ 解决: 隔离/池化   │ 解决: 熔断/降级          │
└─────────────────┴─────────────────┴─────────────────────────┘
```

## 🚦 限流 (Rate Limiting)

### 为什么需要限流？

```
正常请求: 1000 QPS
↓
[系统容量: 5000 QPS] → 正常处理 ✅

突发流量: 50000 QPS（秒杀场景）
↓
[系统容量: 5000 QPS] → 系统崩溃 ❌

有限流保护:
突发流量: 50000 QPS
↓
[限流: 5000 QPS] → [系统: 5000 QPS] → 正常处理 ✅
                  （多余请求被拒绝或排队）
```

### 限流算法对比

| 算法         | 原理           | 优点     | 缺点         | 适用场景 |
| ------------ | -------------- | -------- | ------------ | -------- |
| **计数器**   | 固定窗口内计数 | 实现简单 | 临界问题     | 简单场景 |
| **滑动窗口** | 滑动时间窗口   | 更精确   | 内存占用     | 精确限流 |
| **令牌桶**   | 按速率放入令牌 | 允许突发 | 相对复杂     | API 限流 |
| **漏桶**     | 恒定速率流出   | 流量整形 | 无法应对突发 | 消息队列 |

### 1. 令牌桶算法 (Token Bucket)

```
核心思想：
- 以固定速率向桶中放入令牌
- 请求需要获取令牌才能被处理
- 桶满时，新令牌被丢弃
- 允许一定程度的突发流量

        ┌──────────────┐
        │   令牌生成器   │  速率: r 个/秒
        │   ●●●●●●●    │
        └──────┬───────┘
               ↓ 令牌
        ┌──────────────┐
        │     令牌桶     │  容量: b
        │  ● ● ● ● ●   │
        │  ● ● ● ○ ○   │  ● = 令牌, ○ = 空位
        └──────┬───────┘
               ↓ 获取令牌
        ┌──────────────┐
        │     请求      │
        │  有令牌 → 处理  │
        │  无令牌 → 拒绝  │
        └──────────────┘

特点:
1. 桶内有令牌 → 可以处理请求
2. 桶空了 → 拒绝或等待
3. 支持突发: 桶满时可以一次处理 b 个请求
```

### 2. 滑动窗口算法 (Sliding Window)

```
解决固定窗口的临界问题：

固定窗口问题:
|-------- 窗口1 --------|-------- 窗口2 --------|
                   ↑ 临界点
在 0:59 和 1:01 分别打满限制，实际 2 秒内流量翻倍

滑动窗口解决方案:
|←────── 滑动窗口 ──────→|
     ↑ 任意时刻都是完整的时间窗口

实现方式:
┌────┬────┬────┬────┬────┬────┐
│ Slot│ Slot│ Slot│ Slot│ Slot│ Slot│  每个 slot = 100ms
│  5  │  10 │  8  │  12 │  7  │  0  │  窗口 = 6 slots = 600ms
└────┴────┴────┴────┴────┴────┘
总计: 42 个请求  限制: 60 → 允许更多请求

→ 时间流动，窗口滑动 →
     ┌────┬────┬────┬────┬────┬────┐
     │ 10 │  8  │  12 │  7  │  0  │  15 │
     └────┴────┴────┴────┴────┴────┘
     总计: 52 个请求
```

### 3. 漏桶算法 (Leaky Bucket)

```
核心思想：
- 请求先进入桶中排队
- 以固定速率从桶中流出处理
- 桶满时，新请求被丢弃
- 流量整形，消除突发

        ┌──────────────┐
        │   突发请求    │
        │  ●●●●●●●●●   │  流入可以是突发的
        └──────┬───────┘
               ↓ 入队
        ┌──────────────┐
        │     漏桶      │  容量有限
        │  ●●●●●●      │
        │  ●●●●●●      │  请求在这里排队等待
        │  ●●●         │
        └──────┬───────┘
               ↓ 恒定速率出队
        ┌──────────────┐
        │   处理请求    │  速率固定: r 个/秒
        │  ●   ●   ●   │  无论入队多快，出队恒定
        └──────────────┘

令牌桶 vs 漏桶:
- 令牌桶: 控制平均速率，允许一定突发
- 漏桶: 严格控制速率，完全消除突发
```

## 🔥 熔断 (Circuit Breaker)

### 为什么需要熔断？

```
服务调用链：
用户 → 服务A → 服务B → 服务C → 数据库

服务C 故障时：
没有熔断:
用户 → A(等待) → B(等待) → C(超时) → ❌
       ↓ 线程池耗尽
      A 服务也挂了
       ↓ 级联失败
      整个系统崩溃 💥

有熔断保护:
用户 → A → B → [熔断器: C 已熔断] → 快速失败/降级响应 ✅
              ↓ 不再调用 C
      保护 A、B 服务正常
```

### 熔断器状态机

```
                    +─────────────────────────────────+
                    │                                 │
                    │   ┌───────────────────────┐     │
                    │   │       CLOSED          │     │
                    │   │    (正常状态)          │     │
         成功       │   │   - 调用正常执行       │     │
         ↑          │   │   - 统计失败率        │     │
         │          │   └───────────┬───────────┘     │
         │          │               │                 │
         │          │    失败率 > 阈值 (如 50%)        │
         │          │               │                 │
         │          │               ↓                 │
         │          │   ┌───────────────────────┐     │
         │          │   │        OPEN           │     │
         │          │   │    (熔断状态)          │     │
         │          │   │   - 快速失败           │     │
         │          │   │   - 不调用下游          │     │
         │          │   └───────────┬───────────┘     │
         │          │               │                 │
         │          │    超时（如 30s）                │
         │          │               │                 │
         │          │               ↓                 │
    ┌────┴──────────┴───────────────────────────┐     │
    │            HALF_OPEN                      │     │
    │          (半开状态)                        │     │
    │   - 允许部分请求通过测试                   │     │
    │   - 成功 → CLOSED                         │     │
    │   - 失败 → OPEN                           │─────+
    └───────────────────────────────────────────┘
```

### 熔断参数配置

| 参数       | 描述                   | 推荐值 |
| ---------- | ---------------------- | ------ |
| 失败率阈值 | 触发熔断的失败百分比   | 50%    |
| 慢调用阈值 | 慢调用占比触发熔断     | 100%   |
| 慢调用时间 | 定义慢调用的时间       | 60s    |
| 熔断时长   | OPEN 状态持续时间      | 30s    |
| 最小请求数 | 计算失败率的最小样本   | 10     |
| 半开请求数 | HALF_OPEN 允许的请求数 | 10     |

## 📉 降级 (Degradation)

### 降级策略矩阵

```
┌────────────────┬────────────────────────────────────────────┐
│     降级类型    │                 描述                        │
├────────────────┼────────────────────────────────────────────┤
│   功能降级     │ 关闭非核心功能，保证核心功能                    │
│               │ 例：关闭推荐系统，保留商品列表                   │
├────────────────┼────────────────────────────────────────────┤
│   数据降级     │ 返回缓存/静态/兜底数据                         │
│               │ 例：库存查询失败返回"有货"                      │
├────────────────┼────────────────────────────────────────────┤
│   体验降级     │ 降低服务质量，不关闭服务                        │
│               │ 例：图片从高清改成缩略图                        │
├────────────────┼────────────────────────────────────────────┤
│   异步降级     │ 同步改异步，接收请求后排队处理                   │
│               │ 例：短信验证码排队发送                          │
└────────────────┴────────────────────────────────────────────┘
```

### 降级决策流程

```
                          请求到达
                             │
                             ↓
                    ┌─────────────────┐
                    │  系统健康检查    │
                    └────────┬────────┘
                             │
              ┌──────────────┼──────────────┐
              ↓              ↓              ↓
         健康度高        健康度中        健康度低
              │              │              │
              ↓              ↓              ↓
         正常处理        部分降级        完全降级
              │              │              │
              ↓              ↓              ↓
         完整响应      缓存/简化数据    兜底/静态数据
```

## 🏠 舱壁隔离 (Bulkhead)

### 隔离模式

```
取自泰坦尼克号设计：船体分隔成多个舱室，一个舱室进水不影响其他。

应用到系统设计：

传统设计（无隔离）:
┌─────────────────────────────────────────────┐
│              共享线程池 (200)                │
│  ┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐        │
│  │ API │  │ API │  │ API │  │ API │        │
│  │  A  │  │  B  │  │ C(慢)│  │  D  │        │
│  └─────┘  └─────┘  └─────┘  └─────┘        │
└─────────────────────────────────────────────┘
服务C变慢 → 占满所有线程 → 所有服务不可用 💥

舱壁隔离设计:
┌───────────┬───────────┬───────────┬───────────┐
│  线程池A   │  线程池B   │  线程池C   │  线程池D   │
│   (50)    │   (50)    │   (50)    │   (50)    │
│  ┌─────┐  │  ┌─────┐  │  ┌─────┐  │  ┌─────┐  │
│  │ API │  │  │ API │  │  │ API │  │  │ API │  │
│  │  A  │  │  │  B  │  │  │ C(慢)│  │  │  D  │  │
│  └─────┘  │  └─────┘  │  └─────┘  │  └─────┘  │
└───────────┴───────────┴───────────┴───────────┘
服务C变慢 → 只影响池C → 其他服务正常 ✅
```

### 隔离方式

| 隔离方式       | 描述               | 优点     | 缺点         |
| -------------- | ------------------ | -------- | ------------ |
| **线程池隔离** | 每个服务独立线程池 | 完全隔离 | 线程切换开销 |
| **信号量隔离** | 限制并发数         | 轻量级   | 不能控制超时 |
