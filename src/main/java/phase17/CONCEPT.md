> **核心问题**：如何在分布式环境下保证数据的可靠存储和一致性

## 🎯 分布式存储的核心挑战

### 1. CAP 定理回顾

```
              Consistency (一致性)
                    /\
                   /  \
                  /    \
                 /  CA  \
                /________\
               /\        /\
              /  \  CP  /  \
             / AP \    /    \
            /______\  /______\
    Availability    \/    Partition Tolerance
      (可用性)              (分区容错性)

分布式系统只能同时满足两个:
- CA: 放弃分区容错 (单机数据库)
- CP: 放弃可用性 (ZooKeeper, HBase)
- AP: 放弃强一致性 (Cassandra, DynamoDB)
```

### 2. 数据分布策略

| 策略           | 描述          | 优点               | 缺点               |
| -------------- | ------------- | ------------------ | ------------------ |
| **哈希分片**   | hash(key) % N | 均匀分布           | 扩容需迁移大量数据 |
| **一致性哈希** | 虚拟环形空间  | 扩容只迁移相邻节点 | 热点问题           |
| **范围分片**   | 按范围划分    | 范围查询高效       | 可能不均匀         |
| **目录分片**   | 查表确定位置  | 灵活               | 目录成为瓶颈       |

## 🔄 一致性模型

### 一致性级别对比

```
强一致性 (Strong Consistency)
│  所有节点看到相同的数据，牺牲延迟
│  适用: 金融交易、库存扣减
│
├── 线性一致性 (Linearizability)
│   写操作完成后，所有读都能看到最新值
│
├── 顺序一致性 (Sequential Consistency)
│   所有操作按某个全局顺序执行
│
最终一致性 (Eventual Consistency)
│  允许短暂不一致，最终达到一致
│  适用: 社交动态、商品评论
│
├── 因果一致性 (Causal Consistency)
│   有因果关系的操作保持顺序
│
└── 读己之写 (Read Your Writes)
    写操作后，同一客户端能读到自己的写入
```

### 一致性 vs 可用性 权衡

```
┌─────────────────────────────────────────────────────────────┐
│                    一致性与可用性权衡                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   强一致性                                     最终一致性    │
│      │                                              │      │
│      ▼                                              ▼      │
│   [银行转账]  [库存扣减]  [订单创建]  [商品浏览]  [日志]    │
│      │                                              │      │
│ 延迟高、可用性低                          延迟低、可用性高 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 📊 Raft 共识算法

### Raft 核心概念

Raft 是一种易于理解的分布式共识算法，用于在多个节点间达成数据一致。

```
┌─────────────────────────────────────────────────────────────┐
│                      Raft 节点状态                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Follower ──[超时]──> Candidate ──[获多数票]──> Leader     │
│      ↑                     │                        │      │
│      │                     │                        │      │
│      └─────[发现新Leader]──┴───────────────────────┘      │
│                                                             │
├─────────────────────────────────────────────────────────────┤
│  Follower:   被动接收 Leader 的日志复制                      │
│  Candidate:  发起选举，请求其他节点投票                       │
│  Leader:     处理客户端请求，复制日志到 Follower              │
└─────────────────────────────────────────────────────────────┘
```

### Leader 选举流程

```
1. 初始状态: 所有节点都是 Follower
   [F1] [F2] [F3] [F4] [F5]

2. Follower 超时，变成 Candidate，发起选举
   [F1] [C2] [F3] [F4] [F5]
         │
         └──> 向所有节点发送 RequestVote RPC

3. 获得多数票 (3/5)，成为 Leader
   [F1] [L2] [F3] [F4] [F5]
         │
         └──> 开始发送心跳，维持 Leader 地位

4. Leader 定期发送心跳
   [F1] <─心跳─ [L2] ─心跳─> [F3]
                  │
                  └─心跳─> [F4] [F5]
```

### 日志复制流程

```
Client 请求: SET x = 5

1. Leader 接收请求，写入本地日志
   Leader: [idx:1 term:1 SET x=5]

2. Leader 将日志发送给所有 Follower
   Leader ──AppendEntries──> Follower1, Follower2, ...

3. 多数节点确认后，Leader 提交日志
   Leader: committed = true

4. Leader 通知 Follower 提交
   Leader ──Commit──> Followers

5. 返回客户端成功
   Leader ──OK──> Client
```

### 传统数据库 vs NewSQL

```
┌─────────────┬───────────────────┬───────────────────────────┐
│   特性       │   传统关系数据库   │        NewSQL            │
├─────────────┼───────────────────┼───────────────────────────┤
│   扩展性    │   垂直扩展(加硬件) │   水平扩展(加节点)        │
├─────────────┼───────────────────┼───────────────────────────┤
│   分布式    │   主从复制         │   原生分布式              │
├─────────────┼───────────────────┼───────────────────────────┤
│   事务      │   单机事务         │   分布式事务              │
├─────────────┼───────────────────┼───────────────────────────┤
│   SQL支持   │   完整支持         │   完整支持                │
├─────────────┼───────────────────┼───────────────────────────┤
│   代表产品  │   MySQL, Oracle    │   TiDB, CockroachDB      │
└─────────────┴───────────────────┴───────────────────────────┘
```

### TiDB 架构

```
                    ┌─────────────────────┐
                    │      TiDB Server    │ ← SQL 解析、优化
                    │  (无状态、可水平扩展) │
                    └──────────┬──────────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
              ▼                ▼                ▼
        ┌──────────┐    ┌──────────┐    ┌──────────┐
        │  TiKV-1  │    │  TiKV-2  │    │  TiKV-3  │
        │  Region  │    │  Region  │    │  Region  │
        │ [1-100]  │    │[101-200] │    │[201-300] │
        └──────────┘    └──────────┘    └──────────┘
              ↑                ↑                ↑
              └────────────────┼────────────────┘
                               │
                    ┌──────────┴──────────┐
                    │         PD          │ ← 元数据管理、调度
                    │ (Placement Driver)  │
                    └─────────────────────┘

TiDB: SQL 层，解析 SQL，生成执行计划
TiKV: 存储层，分布式 KV 存储，Raft 复制
PD:   调度层，元数据管理，Region 调度
```

## 💾 LSM-Tree 存储引擎

### LSM-Tree 结构

```
┌─────────────────────────────────────────────────────────────┐
│                     LSM-Tree 结构                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   写入                                                      │
│     │                                                       │
│     ▼                                                       │
│  ┌──────────────┐                                           │
│  │   MemTable   │ ← 内存中的有序数据结构 (红黑树/跳表)        │
│  └──────┬───────┘                                           │
│         │ 刷盘                                              │
│         ▼                                                   │
│  ┌──────────────┐                                           │
│  │ Immutable    │ ← 不可变，等待写入磁盘                     │
│  │  MemTable    │                                           │
│  └──────┬───────┘                                           │
│         │                                                   │
│         ▼                                                   │
│  ┌──────────────────────────────────────────────┐          │
│  │                    磁盘                       │          │
│  │  Level 0: [SST] [SST] [SST]     (可能重叠)   │          │
│  │  Level 1: [  SST  ] [  SST  ]   (有序不重叠) │          │
│  │  Level 2: [    SST    ] [    SST    ]       │          │
│  │  Level 3: [        SST        ]             │          │
│  └──────────────────────────────────────────────┘          │
│                                                             │
└─────────────────────────────────────────────────────────────┘

写放大: 数据可能被多次写入 (Compaction)
读放大: 可能需要查询多个 Level
空间放大: 数据在多个 Level 暂时共存
```

### 写入与读取流程

```
写入流程 (顺序写，高性能):
1. 写入 WAL (预写日志，崩溃恢复)
2. 写入 MemTable (内存)
3. MemTable 满了 → 变成 Immutable
4. 后台异步刷盘为 SSTable
5. 后台 Compaction 合并 SSTable

读取流程:
1. 查 MemTable (内存)
2. 查 Immutable MemTable
3. 查 Level 0 SSTable (可能多个)
4. 查 Level 1, 2, 3... (二分查找)
5. 布隆过滤器加速判断 key 是否存在
```

## 🏗️ 数据架构模式

### 数据湖 vs 数据仓库

```
┌─────────────────────────────────────────────────────────────┐
│                   数据湖 (Data Lake)                         │
├─────────────────────────────────────────────────────────────┤
│  特点: 存储原始数据，Schema-on-Read                          │
│  格式: 任意格式 (JSON, CSV, Parquet, 图片, 视频)             │
│  用途: 探索性分析、机器学习                                  │
│  技术: HDFS, S3, Delta Lake                                 │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                  数据仓库 (Data Warehouse)                   │
├─────────────────────────────────────────────────────────────┤
│  特点: 存储清洗后的数据，Schema-on-Write                     │
│  格式: 结构化数据，预定义 Schema                             │
│  用途: BI 报表、固定分析                                    │
│  技术: Snowflake, BigQuery, ClickHouse                     │
└─────────────────────────────────────────────────────────────┘

趋势: 湖仓一体 (Lakehouse) = 数据湖 + 数据仓库
```
