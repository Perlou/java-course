## 📊 性能指标体系

### 核心指标

| 指标           | 英文                 | 含义                       | 目标值示例  |
| -------------- | -------------------- | -------------------------- | ----------- |
| **响应时间**   | Response Time (RT)   | 请求从发出到收到响应的时间 | P99 < 200ms |
| **吞吐量**     | Throughput (TPS/QPS) | 单位时间处理的请求数       | > 10000 QPS |
| **并发数**     | Concurrency          | 同时处理的请求数           | > 1000      |
| **错误率**     | Error Rate           | 失败请求的比例             | < 0.1%      |
| **资源利用率** | Utilization          | CPU/内存/IO 使用率         | 70-80%      |

### 响应时间百分位

```java
/**
 * 响应时间统计示例
 *
 * P50 = 50ms   → 50% 的请求在 50ms 内完成
 * P90 = 100ms  → 90% 的请求在 100ms 内完成
 * P99 = 200ms  → 99% 的请求在 200ms 内完成
 * P999 = 500ms → 99.9% 的请求在 500ms 内完成
 *
 * ⚠️ 关注 P99，而不仅仅是平均值！
 * 平均值可能掩盖长尾问题
 */
```

### GC 类型对比

| GC 收集器      | 特点                 | 适用场景         | 停顿时间 |
| -------------- | -------------------- | ---------------- | -------- |
| **Serial**     | 单线程               | 客户端应用       | 高       |
| **Parallel**   | 多线程，吞吐量优先   | 批处理任务       | 中       |
| **CMS**        | 并发标记清除         | 低延迟应用       | 低       |
| **G1**         | 分区收集，可预测停顿 | 大堆、平衡场景   | 可控     |
| **ZGC**        | 低延迟，TB 级堆      | 超大堆、极低延迟 | < 10ms   |
| **Shenandoah** | 并发压缩             | 低延迟需求       | < 10ms   |

### 常用 JVM 参数

```bash
# 堆内存配置
-Xms4g -Xmx4g          # 初始/最大堆内存（建议设置相同值）
-Xmn2g                  # 年轻代大小
-XX:MetaspaceSize=256m  # 元空间初始大小

# G1 收集器
-XX:+UseG1GC
-XX:MaxGCPauseMillis=200     # 目标停顿时间
-XX:G1HeapRegionSize=16m     # Region 大小

# ZGC (JDK 15+)
-XX:+UseZGC
-XX:+ZGenerational           # JDK 21+ 分代 ZGC

# GC 日志
-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=100m

# 调试参数
-XX:+HeapDumpOnOutOfMemoryError
-XX:HeapDumpPath=/path/to/dump
```

## 💾 SQL 优化核心概念

### 执行计划分析

```sql
-- EXPLAIN 关键字段说明
EXPLAIN SELECT * FROM orders WHERE user_id = 100;

/*
┌────────────────────────────────────────────────────────────┐
│ id │ select_type │ table  │ type  │ key     │ rows │ Extra │
├────────────────────────────────────────────────────────────┤
│ 1  │ SIMPLE      │ orders │ ref   │ idx_uid │ 50   │       │
└────────────────────────────────────────────────────────────┘

type 字段（访问类型，从好到差）：
  system > const > eq_ref > ref > range > index > ALL

⚠️ 目标：避免 ALL（全表扫描），争取 ref 或更好
*/
```

### 索引优化原则

```
最左前缀原则
═══════════════════════════════════════════════════════════

索引: (a, b, c)

✅ 可以使用索引:
   WHERE a = 1
   WHERE a = 1 AND b = 2
   WHERE a = 1 AND b = 2 AND c = 3

❌ 无法使用索引:
   WHERE b = 2              -- 缺少最左列 a
   WHERE a = 1 AND c = 3    -- 跳过了 b

⚠️ 范围条件后的列无法使用索引:
   WHERE a > 1 AND b = 2    -- b 无法使用索引
```

## 🗄️ 缓存策略

### 缓存架构

```
┌────────────┐    ┌────────────┐    ┌────────────┐    ┌────────────┐
│   客户端    │ → │   本地缓存  │ → │  分布式缓存 │ → │   数据库    │
│            │    │  (Caffeine) │    │   (Redis)  │    │  (MySQL)   │
└────────────┘    └────────────┘    └────────────┘    └────────────┘
                        ↓                  ↓                 ↓
                    ~1μs             ~1ms              ~10ms
                   (微秒级)          (毫秒级)           (毫秒级)
```

### 缓存问题与解决方案

| 问题           | 描述              | 解决方案                    |
| -------------- | ----------------- | --------------------------- |
| **缓存穿透**   | 查询不存在的数据  | 布隆过滤器、空值缓存        |
| **缓存击穿**   | 热点 key 失效     | 互斥锁、永不过期 + 异步更新 |
| **缓存雪崩**   | 大量 key 同时失效 | 随机过期时间、多级缓存      |
| **一致性问题** | 缓存与 DB 不一致  | Cache-Aside、延迟双删       |

### 缓存更新策略

```java
/**
 * Cache-Aside 模式（旁路缓存）
 *
 * 读流程:
 * 1. 先查缓存
 * 2. 缓存命中 → 直接返回
 * 3. 缓存未命中 → 查 DB → 写入缓存 → 返回
 *
 * 写流程:
 * 1. 更新 DB
 * 2. 删除缓存（而非更新缓存！）
 *
 * ⚠️ 为什么删除而不是更新？
 * - 避免并发更新导致的数据不一致
 * - 懒加载思想，需要时再加载
 */
```

## 🔗 连接池调优

### 连接池参数

```yaml
# HikariCP (最快的连接池) 推荐配置
spring:
  datasource:
    hikari:
      # 连接池大小 = ((CPU核心数 * 2) + 磁盘数)
      # 一般 10-20 就够了！
      maximum-pool-size: 10
      minimum-idle: 10

      # 连接超时
      connection-timeout: 30000 # 30s
      idle-timeout: 600000 # 10min
      max-lifetime: 1800000 # 30min

      # 连接验证
      connection-test-query: SELECT 1
```

### 连接池大小公式

```
最佳连接数 ≈ (核心数 × 2) + 有效磁盘数

示例：
- 4 核 CPU + 1 个 SSD
- 最佳连接数 = (4 × 2) + 1 = 9

⚠️ 更多的连接不一定更好！
- 连接过多 → 上下文切换开销增加
- 连接过少 → 请求排队等待
- 需要根据实际压测调整
```

## 📈 性能测试方法论

### 测试类型

| 类型           | 目的         | 方法                 |
| -------------- | ------------ | -------------------- |
| **基准测试**   | 建立性能基线 | 固定负载，记录指标   |
| **负载测试**   | 验证预期负载 | 模拟正常流量         |
| **压力测试**   | 找到系统极限 | 逐步增加负载直到崩溃 |
| **稳定性测试** | 验证长期运行 | 持续运行 24h+        |
| **峰值测试**   | 验证突发流量 | 模拟秒杀等场景       |

## 📝 学习建议

1. **先测量，再优化** - 没有数据支撑的优化是盲目的
2. **二八法则** - 80% 的性能问题集中在 20% 的代码
3. **一次只改一个变量** - 便于验证效果
4. **警惕过早优化** - 先保证正确性，再考虑性能
5. **建立监控体系** - 线上问题依赖监控发现

> 💡 "过早优化是万恶之源" —— Donald Knuth
