## 📋 技术债务管理

### 技术债务类型

```
┌─────────────────────────────────────────────────────────────┐
│                    技术债务分类                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   ┌─────────────┐  ┌─────────────┐  ┌─────────────┐        │
│   │  故意债务   │  │  无意债务   │  │  过时债务   │        │
│   │             │  │             │  │             │        │
│   │ 快速上线    │  │ 设计缺陷    │  │ 技术演进    │        │
│   │ 临时方案    │  │ 经验不足    │  │ 依赖老化    │        │
│   └─────────────┘  └─────────────┘  └─────────────┘        │
│                                                             │
│   影响评估:                                                 │
│   ┌────────────────────────────────────────────────────┐   │
│   │  债务成本 = 每次修改的额外时间 × 预计修改次数       │   │
│   │  偿还收益 = 债务成本 - 偿还投入                     │   │
│   └────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 债务偿还策略

| 策略           | 描述                     | 适用场景           |
| -------------- | ------------------------ | ------------------ |
| **童子军原则** | 每次修改让代码更好一点   | 日常开发           |
| **专项重构**   | 分配专门时间处理         | 积累较多的债务     |
| **重写替换**   | 构建新系统逐步替换旧系统 | 债务过重，难以维护 |
| **隔离封装**   | 不偿还，但隔离影响范围   | 低优先级或即将下线 |

## 📝 架构决策记录 (ADR)

### ADR 模板

```markdown
# ADR-001: 使用 Kafka 作为消息队列

## 状态

已采纳 (Accepted)

## 背景

系统需要异步处理订单，解耦服务间依赖，削峰填谷。

## 决策

选择 Apache Kafka 作为消息队列中间件。

## 理由

1. 高吞吐量：支持百万级 TPS
2. 持久化：消息不丢失，可重放
3. 团队经验：已有运维经验
4. 生态完善：与 Spring、Flink 集成良好

## 备选方案

- RabbitMQ：功能丰富但吞吐量较低
- RocketMQ：国产，但社区较小

## 后果

- 需要部署 ZooKeeper 集群
- 消费者需要处理消息幂等性
- 增加运维复杂度

## 相关决策

- ADR-002: Kafka 消费者幂等性设计
```
