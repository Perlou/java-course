## 🔑 OAuth 2.0 深入

### 授权流程对比

```
┌─────────────────────────────────────────────────────────────┐
│                    OAuth 2.0 授权流程                        │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│ 授权码模式 (Authorization Code) - 最安全                    │
│ ┌──────────────────────────────────────────────────────┐   │
│ │ 用户 → 应用 → 认证服务器 → 用户授权                    │   │
│ │                   ↓                                  │   │
│ │              授权码 (code)                           │   │
│ │                   ↓                                  │   │
│ │ 应用 → 认证服务器 (code + secret) → Access Token      │   │
│ └──────────────────────────────────────────────────────┘   │
│ 适用场景: Web 应用后端                                      │
│                                                             │
│ PKCE 增强 (Proof Key for Code Exchange)                     │
│ ┌──────────────────────────────────────────────────────┐   │
│ │ 应用生成 code_verifier → hash → code_challenge        │   │
│ │ 授权请求携带 code_challenge                           │   │
│ │ Token 请求携带 code_verifier (证明是同一客户端)        │   │
│ └──────────────────────────────────────────────────────┘   │
│ 适用场景: SPA、移动应用（无法安全存储 secret）               │
│                                                             │
│ 客户端凭证模式 (Client Credentials)                         │
│ ┌──────────────────────────────────────────────────────┐   │
│ │ 应用 → 认证服务器 (client_id + secret) → Access Token │   │
│ └──────────────────────────────────────────────────────┘   │
│ 适用场景: 服务间调用（无用户参与）                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### OAuth 2.0 vs OpenID Connect

| 特性     | OAuth 2.0            | OIDC                    |
| -------- | -------------------- | ----------------------- |
| 目的     | 授权 (Authorization) | 认证 (Authentication)   |
| 返回内容 | Access Token         | Access Token + ID Token |
| 用户信息 | 需额外请求           | ID Token 内置用户信息   |
| 标准化   | 授权框架             | OAuth2 之上的认证协议   |

## 🛡️ 权限设计模型

### RBAC vs ABAC

```
┌─────────────────────────────────────────────────────────────┐
│              RBAC (Role-Based Access Control)               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   用户 ──→ 角色 ──→ 权限                                    │
│                                                             │
│   ┌──────┐    ┌──────┐    ┌─────────────────────┐          │
│   │ 张三  │───▶│ Admin │───▶│ user:read          │          │
│   └──────┘    └──────┘    │ user:write         │          │
│                            │ order:read         │          │
│   ┌──────┐    ┌──────┐    │ order:write        │          │
│   │ 李四  │───▶│ User  │───▶│ order:delete      │          │
│   └──────┘    └──────┘    └─────────────────────┘          │
│                                                             │
│   优点: 简单直观，易于管理                                  │
│   缺点: 权限粒度粗，无法处理复杂场景                        │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│           ABAC (Attribute-Based Access Control)             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   决策 = f(主体属性, 资源属性, 环境属性, 操作)               │
│                                                             │
│   示例规则:                                                 │
│   "部门经理可以在工作时间内查看本部门员工工资"               │
│                                                             │
│   ┌──────────────────────────────────────────────────────┐ │
│   │ 主体属性: role=manager, department=销售               │ │
│   │ 资源属性: type=salary, department=销售                │ │
│   │ 环境属性: time=工作时间, location=公司网络             │ │
│   │ 操作: read                                           │ │
│   │ 结果: ALLOW                                          │ │
│   └──────────────────────────────────────────────────────┘ │
│                                                             │
│   优点: 细粒度控制，灵活                                    │
│   缺点: 规则复杂，性能开销大                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 权限模型对比

| 模型 | 粒度 | 复杂度 | 适用场景             |
| ---- | ---- | ------ | -------------------- |
| ACL  | 细   | 高     | 文件系统             |
| RBAC | 中   | 低     | 企业应用、后台管理   |
| ABAC | 细   | 高     | 合规要求高、复杂策略 |

## 🔒 零信任架构

### 核心原则

```
┌─────────────────────────────────────────────────────────────┐
│                    零信任架构原则                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   传统模型: 边界安全                                        │
│   ┌──────────────────────────────────────────────────┐     │
│   │ 防火墙                                           │     │
│   │    ↓                                             │     │
│   │  [内网: 信任所有]  ←──X──→  [外网: 不信任]        │     │
│   └──────────────────────────────────────────────────┘     │
│   问题: 一旦突破边界，内部无防护                            │
│                                                             │
│   零信任模型: Never Trust, Always Verify                    │
│   ┌──────────────────────────────────────────────────┐     │
│   │                                                  │     │
│   │   每次访问 → 验证身份 → 检查设备 → 评估风险       │     │
│   │              → 最小权限授权 → 持续监控            │     │
│   │                                                  │     │
│   └──────────────────────────────────────────────────┘     │
│                                                             │
│   核心思想:                                                 │
│   1. 永不信任，始终验证                                     │
│   2. 最小权限原则                                          │
│   3. 假设已被攻破                                          │
│   4. 微分段隔离                                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## ⚠️ OWASP Top 10 安全风险

### 2023 十大安全风险

| 排名 | 风险                      | 防御措施                       |
| ---- | ------------------------- | ------------------------------ |
| 1    | Broken Access Control     | RBAC、权限校验、默认拒绝       |
| 2    | Cryptographic Failures    | HTTPS、强加密算法、密钥管理    |
| 3    | Injection (SQL/XSS)       | 参数化查询、输入验证、输出编码 |
| 4    | Insecure Design           | 威胁建模、安全设计评审         |
| 5    | Security Misconfiguration | 安全基线、自动化检查           |
| 6    | Vulnerable Components     | 依赖扫描、及时更新             |
| 7    | Auth Failures             | MFA、密码策略、Session 管理    |
| 8    | Data Integrity Failures   | 签名验证、完整性校验           |
| 9    | Logging & Monitoring      | 安全日志、入侵检测             |
| 10   | SSRF                      | URL 白名单、网络隔离           |

### SQL 注入防御示例

```java
// ❌ 危险: 字符串拼接
String sql = "SELECT * FROM users WHERE name = '" + username + "'";

// ✅ 安全: 参数化查询
PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE name = ?");
ps.setString(1, username);

// ✅ 安全: ORM 框架
userRepository.findByUsername(username);
```
